# Agentic 报告脚本 Building Blocks

当用户要做“日报、周报、监控简报、调研摘要、分析报告、增量报告、跨来源信息聚合”时加载本文件。

最佳实践参考脚本：`assets/script/moore_threads_daily_briefing.py`。该脚本是“目标实体动态报告”的一个实例，里面的模块可以迁移到政策跟踪、产品监控、运营周报、竞品分析、技术巡检、安全简报、文档变更总结等场景。

这类脚本不是“抓取结果拼列表”，而是把一次报告任务拆成可组合的 agentic workflow building blocks：

```text
领域配置 → 结构化模型 → 状态库 → 权威锚点 → 信息发现 → 候选筛选
→ 内容获取 → 原子事实抽取 → 时效过滤 → 信息单元聚合 → 状态增强
→ 事实边界检查 → 一屏摘要 → 长报告 → 短消息 → 成功后写状态
```

## 适用场景

- 定期生成某个目标实体、主题、项目、文档、系统、市场或风险域的报告。
- 需要跨次运行去重，区分冷启动基线、新增信息和延续信息。
- 输入来源多且质量不一，需要先获取正文或原始数据，再做事实抽取。
- 同一事实或事件会被多个来源重复出现，需要聚合成信息单元而不是罗列原始条目。
- 需要短 `submit_result.message` 和长 Markdown 报告分离交付。
- 需要保留来源、证据、置信度和后续关注点。

## Block 1：领域配置

参考逻辑：

1. 判断脚本是“通用工具”还是“专用任务”。如果用户明确要长期跟踪固定对象，优先做专用脚本。
2. 专用脚本把目标实体、主题范围、报告标题、输出目录、状态库路径写成常量。
3. 把搜索窗口、采集数量、正文长度、LLM 并发、报告明细数量也写成常量。
4. 所有预算上限都要有业务含义：候选上限控制噪声，内容获取上限控制工具成本，正文截断上限控制 LLM 输入预算。

输入：

- 目标实体或主题：公司、行业、政策、项目、系统、文档、产品、用户群体等。
- 报告频率：一次性、日报、周报、定时监控。
- 关注维度：变化、风险、机会、异常、指标、竞品、用户反馈、文档更新等。

输出：

- 一组清晰常量。
- 一个固定的输出路径。
- 一个固定的状态库路径。

不要做：

- 不要为了“灵活”给专用日报保留大量 CLI 入参。
- 不要让运行时参数决定业务对象，否则状态库会混入多个对象的数据。
- 不要无限制搜索、fetch 或传入全文。

## Block 2：结构化模型

参考逻辑：

1. 先把流程拆成多个 LLM 阶段，每个阶段只负责一个决策。
2. 为每个阶段定义一个 Pydantic `BaseModel`。
3. 模型字段只保留下游真正会用的内容，不要把“解释原因”“全文摘要”等无约束文本一路传递。
4. 每个模型都设置安全默认值，保证 LLM 返回缺字段时脚本还能降级执行。

推荐字段边界：

- 搜索规划：`queries`、`reason`。
- 候选筛选：`selected_urls`、`source_assessments`。
- 事实抽取：`category`、`title`、`content`、`source_url`、`publish_date`、`source_tier`、`claim_strength`、`fact_boundary`。
- 信息单元聚合：`item_title`、`publish_date`、`importance`、`confidence`、`why_it_matters`、`impact`、`evidence`、`source_urls`。
- 用户摘要：`headline`、`overall_summary`、`key_changes`、`risk_flags`、`watch_points`。

输出：

- 每一阶段都产出结构化对象，而不是散文本。

不要做：

- 不要让最终报告 LLM 直接读取原始搜索结果。
- 不要把完整 HTML、完整 JSON 或大段日志放入结构化模型字段。
- 不要用一个超大 response_format 同时完成搜索筛选、事实抽取、聚合和报告。

## Block 3：状态库

参考逻辑：

1. 启动时初始化 SQLite。
2. 先读取历史信息单元标题、历史快照和运行记录。
3. 运行过程中只读状态，不提前写状态。
4. 生成报告成功后，再一次性写入来源、信息单元、报告快照和本次运行记录。
5. 失败路径不写状态，避免某次失败把未展示内容标成已处理。

推荐表设计：

- `seen_sources`：来源级去重，来源可以是 URL、文件路径、文档 token、API item id。
- `items`：信息单元级去重，用于合并同一事件、同一变更、同一异常或同一指标说明。
- `daily_items` / `report_items`：每次报告快照。
- `runs`：运行记录和冷启动判断。

关键判断：

- 如果 `runs` 中没有早于本次的记录，本次是冷启动 baseline。
- baseline 模式下，信息可以入库，但报告不能把历史存量写成新增发现。
- 如果来源已见但标题略变，应交给信息单元级匹配判断是否延续。
- 如果来源未见但信息单元与历史高度相似，应标为 continuing。

输出：

- 每个信息单元获得 `lifecycle`: `baseline` / `new` / `continuing`。
- 状态库中保存本次报告路径、信息单元数、新增数。

不要做：

- 不要只靠 URL 去重。
- 不要首次运行就把所有搜到的历史信息标为新增。
- 不要在 fetch 成功或抽取成功后立刻写状态。

## Block 4：权威锚点

参考逻辑：

1. 先找权威锚点：官方 API、数据库、固定网页、内部系统、用户提供文件、监控指标、文档快照。
2. 用权威锚点提供报告基准，例如核心指标、最新状态、版本号、公告日期、任务结果、当前库存、服务健康度。
3. 再用搜索结果、网页、日志、文档或其他来源补充解释。
4. 如果主源失败，切到兜底源；如果全部失败，返回带 `error` 的占位结构。

输入：

- 固定 API URL、固定网页、数据库查询、文件路径、文档 token 或监控接口。
- 请求参数、headers、timeout。

处理步骤：

- 请求主源。
- 解析成统一字段结构。
- 请求兜底源。
- 仍失败时返回 `is_available=false` 或 `is_trading_day=false` 这类占位字段。

输出：

- 一个标准 dict，供 prompt、报告表格和口径检查复用。

不要做：

- 不要用搜索 snippet 当确定数据。
- 不要让主源失败直接中断整个报告，除非该数据是任务唯一目标。
- 不要让主源和兜底源返回不同字段结构。

## Block 5：信息发现规划

参考逻辑：

1. 准备基础检索线索，覆盖稳定主题。
2. 从状态库读取最近信息单元标题、风险点、异常点或关注点。
3. 把基础线索和历史上下文交给 LLM，让它补充少量动态检索线索。
4. 合并基础线索和动态线索，去重、限量。

输入：

- `BASE_QUERIES`、固定 URL、API 参数、文件路径或文档 token
- 最近已跟踪信息单元标题
- 用户关注主题

Prompt 要点：

- 要求覆盖固定主题。
- 要求结合近期信息单元扩展。
- 要求不要生成太泛的词。
- 要求最多返回 N 个 query。

输出：

- 搜索 query、固定 URL、API 参数或文件处理列表。

降级策略：

- LLM 扩展失败时只使用基础线索。

不要做：

- 不要长期只用固定线索。
- 不要让 LLM 生成几十个检索项。
- 不要把年份日期硬编码进 query 来替代时间筛选。

## Block 6：候选收集

参考逻辑：

1. 根据任务类型调用搜索、fetch、API、文件读取或 CLI。
2. 如果使用搜索，使用 `publish_time` 参数限制时间窗口。
3. 用 `asyncio.Semaphore` 控制独立 I/O 或 SDK 调用并发。
4. 合并结果后按来源 ID 去重。
5. 截断到最大候选数。

输入：

- 检索线索列表。
- 起止时间或版本范围。
- 工具 schema version 或 API 参数。

输出：

- 候选结果列表，包含 title、source_id、source_url/path、publish_time/update_time、snippet/metadata。

降级策略：

- 单个候选收集失败只打印日志，不影响其他候选。
- 全部收集失败时后续阶段应返回空报告或错误提交。

不要做：

- 不要把候选摘要直接传给最终总结。
- 不要无限并发。
- 不要忽略时间范围、版本范围或状态范围。

## Block 7：候选筛选与来源判断

参考逻辑：

1. 将候选整理成编号列表，但要求 LLM 输出完整来源 ID 或 URL。
2. 让 LLM 判断哪些候选值得进入正文/原始数据获取。
3. 同时让 LLM 对来源质量做动态判断，而不是在代码里写白名单。
4. 用规范化来源 ID 或 URL 将 LLM 输出映射回原候选。
5. 把 `source_tier` 和 `source_assessment` 写回候选 dict，传给正文抽取阶段。

输入：

- 候选 title、URL、发布时间、snippet。

输出：

- 需要获取正文或原始数据的候选列表。
- 每个候选携带来源判断。

source tier 可按领域调整，通用建议：

- `official`
- `trusted_third_party`
- `internal`
- `community`
- `unverified`

降级策略：

- LLM 筛选失败时取前 N 条候选，但标记来源为 `unverified`。

不要做：

- 不要只让 LLM 返回序号。
- 不要把来源到 tier 的映射写死在代码里，除非用户明确提供可信源清单。
- 不要让 snippet 进入最终报告事实层。

## Block 8：内容获取

参考逻辑：

1. 只处理筛选后的候选。
2. 每个候选独立获取正文、原始 JSON、文件内容、日志片段或文档内容。
3. 折叠 HTML、空白、转义字符或无关元数据，得到可抽取文本。
4. 保留 title、URL、publish_time、source tier、source assessment。
5. 对正文做输入预算控制。

输入：

- 筛选后的候选。
- fetch tool schema version、API 参数、文件路径或 CLI 参数。

输出：

- 内容列表：title、source_id、source_url/path、publish_time/update_time、source_tier、source_assessment、content。

正文截断原则：

- 可以截断，但必须确认目标信息通常位于前段。
- 截断是预算控制，不是事实过滤。
- 报告仍保留 URL，便于追溯。

不要做：

- 不要处理所有候选。
- 不要丢掉来源判断。
- 不要把单个来源失败当成整体失败，除非没有任何来源成功。

## Block 9：事实抽取

参考逻辑：

1. 每个内容源独立进入事实抽取。
2. Prompt 要求只抽与目标对象和任务主题相关的可核验事实、变化、异常、指标或结论。
3. 每个事实必须带来源 ID 或 URL，以及日期。
4. 同时要求 LLM 判断来源等级、claim 强度和事实边界。
5. `has_new_info=false` 的内容源整项丢弃。

输入：

- 获取后的正文、JSON 摘要、日志片段、文件内容或文档内容。
- 候选阶段来源判断。

输出：

- 原子事实列表。

事实字段建议：

- `category`
- `title`
- `content`
- `source_url`
- `publish_date`
- `source_tier`
- `claim_strength`
- `fact_boundary`

降级策略：

- 单页抽取失败返回空事实，不中断整体。

不要做：

- 不要让最终报告基于原始内容直接写。
- 不要在代码里用关键词判断“强 claim”。
- 不要把无关内容硬塞进信息单元聚合。

## Block 10：时效过滤

参考逻辑：

1. 将 run_date 作为基准日期。
2. 对事实的 publish_date 做解析。
3. 明确早于窗口的事实不上屏。
4. 日期不明的事实可以保留，但后续聚合和报告要降低确定性。

输入：

- 原子事实列表。
- run_date。
- 时间窗口，例如 7 天。

输出：

- 保留的近期事实。

不要做：

- 不要把历史背景当本次新增。
- 不要只依赖搜索工具的 publish_time，fetch 后仍要检查正文日期或 LLM 抽取日期。

## Block 11：信息单元聚合

参考逻辑：

1. 将近期事实整理为 JSON payload。
2. 把权威锚点摘要也传入，作为口径基准。
3. 把历史已跟踪信息单元传入，帮助 LLM 区分新增和延续语义。
4. 要求 LLM 将同一信息、同一变更、同一异常、同一指标解释或同一主题证据合并。
5. 要求每个信息单元输出“为什么重要”和“影响”，而不是复述来源内容。

输入：

- 原子事实列表。
- 权威锚点摘要。
- 历史信息单元标题。

输出：

- 聚合信息单元列表。

信息单元字段建议：

- `item_title`
- `publish_date`
- `category`
- `signal_type`
- `importance`
- `confidence`
- `why_it_matters`
- `impact`
- `evidence`
- `source_urls`
- `source_titles`
- `fact_boundary`
- `conflict_note`

降级策略：

- 聚合 LLM 失败时，将单条事实降级成信息单元，但要明确“尚未完成多来源聚合”。

不要做：

- 不要按来源 ID、URL 或标题逐条罗列。
- 不要让标题变成来源标题合集。
- 不要丢掉来源 ID 或 URL。

## Block 12：状态增强

对应函数：

- `enrich_event_state()`
- `event_fingerprint()`
- `normalize_text()`

参考逻辑：

1. 对每个信息单元生成弱归一化标题。
2. 在历史信息单元库中查找相同或相似项。
3. 同时检查所有来源 ID 是否已见。
4. 根据 baseline、历史匹配和来源状态标记生命周期。
5. 给信息单元写入稳定 key。

生命周期规则：

- baseline 模式：所有信息单元都是 `baseline`，`is_new=false`。
- 非 baseline 且无历史匹配、来源未全见：`new`。
- 否则：`continuing`。

输出：

- 每个信息单元带 `item_key`、`lifecycle`、`is_new`。

不要做：

- 不要写业务词替换表来“归一化”。
- 不要只用标题前 N 字判断同一信息单元。
- 不要把历史延续信息写成新增。

## Block 13：事实边界与口径检查

参考逻辑：

1. 对信息单元做结构兜底：缺失来源等级、claim 强度、置信度时填默认值。
2. 用权威锚点检查口径，例如指标值、版本号、状态、时间、库存、任务结果、行情涨跌。
3. 如果信息单元中的描述和权威锚点冲突，添加 `conflict_note`。
4. 把事实边界和口径提示传到报告，不在代码中强行删除信息单元。

输入：

- 聚合信息单元。
- 权威锚点。

输出：

- 口径提示列表。
- 带 fact boundary 的信息单元。

不要做：

- 不要在代码里硬编码来源等级。
- 不要在代码里硬编码强 claim 关键词。
- 不要把冲突事实静默丢弃，应提示核验口径。

## Block 14：一屏摘要

参考逻辑：

1. 将聚合信息单元压缩成摘要输入行。
2. 输入行要包含 lifecycle、importance、confidence、source tier 和信息单元判断。
3. 告诉 LLM 当前是否 baseline 模式。
4. 要求输出固定字段：headline、overall_summary、key_changes、impact_assessment、risk_flags、watch_points。

输入：

- 权威锚点摘要。
- 聚合信息单元。
- 历史信息单元参考。
- 口径提示。
- baseline 状态。

输出：

- 面向用户的一屏摘要。

表达约束：

- 使用领域中性的判断，例如“整体偏正面/偏中性/偏风险/信息不足”“对用户体验有改善”“对稳定性有风险”“对后续交付有阻塞”。
- 不要写超出事实的强结论，例如“必然改善”“确定失败”“一定增长”。
- 风险和后续关注必须可跟踪，例如公告、订单、任务状态、错误率、版本发布、用户反馈、资源用量、审批节点。

降级策略：

- 摘要 LLM 失败时，用前 3 个信息单元标题和口径提示生成保守摘要。

## Block 15：长报告渲染

参考逻辑：

1. 报告开头写标题、日期、生成时间。
2. 第一屏写用户摘要，不先堆明细。
3. 写权威锚点表格，例如指标表、状态表、版本表、行情表、任务结果表。
4. 写风险与后续关注。
5. 按业务分类输出信息单元解读。
6. 明细表中建议包含状态、日期、标题、重要性、来源等级、置信度、判断、证据、来源。
7. 报告末尾写完整来源列表。

输入：

- run_date。
- 权威锚点。
- 一屏摘要。
- 聚合信息单元。
- 口径提示。
- baseline 状态。

输出：

- Markdown 字符串，写入 `./codeact/output/`。

Markdown 注意事项：

- 表格 cell 中的 `|` 要替换。
- 多行判断可用 `<br>`。
- 长正文不要放进 `submit_result.message`。
- 本地报告文件名不要有空格。

不要做：

- 不要把原始候选逐条平铺成列表。
- 不要没有来源列表。
- 不要在冷启动报告里写“新增发现”。

## Block 16：短消息交付

参考逻辑：

1. 从一屏摘要中抽取 headline、前 3 条 key changes、impact assessment。
2. 加入权威锚点的最关键指标或状态。
3. 加入信息单元总数、新增数或 baseline 提示。
4. 如果有口径提示，最多放 1-2 条。
5. 用 `computer://` 绝对路径链接到完整报告文件。

输入：

- run_date。
- report_path。
- 权威锚点。
- 一屏摘要。
- 聚合信息单元。
- 口径提示。

输出：

- `submit_result.message` 字符串。

格式要求：

- message 人类可读。
- 本地文件链接格式：`[完整报告](computer://绝对路径)`。
- 网络 URL 不使用 `computer://`。
- 不使用 `![](computer://...)`。

不要做：

- 不要把完整 Markdown 报告塞进 message。
- 不要把错误栈、HTML、JSON 放进 message。

## Block 17：主流程编排

对应函数：

- `main()`

参考逻辑：

```text
初始化 SDK/状态库
→ 判断 baseline
→ 获取权威锚点
→ 生成动态检索线索
→ 收集候选
→ 筛选候选
→ 获取内容
→ 抽取事实
→ 聚合信息单元
→ 匹配历史状态
→ 检查口径冲突
→ 生成摘要和报告
→ 写报告文件
→ 成功后写状态
→ submit_result
```

执行规则：

- 每个阶段都打印简短进度日志。
- 所有成功、失败路径必须 `submit_result`。
- 错误提交使用 `result_mode="notify"`。
- `finally` 中关闭状态库连接。

成功路径：

1. 写报告文件。
2. 写状态库。
3. `submit_result(status="success")`，message 使用短摘要，data 返回报告路径、状态库路径、信息单元数量。

失败路径：

1. 打印 traceback 到日志。
2. `submit_result(result_mode="notify", status="error")`。
3. message 只写简短人类可读原因。
4. 不写状态库。
